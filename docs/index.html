Write AI coding agent hooks once. Run them everywhere.

polyhook detects which AI tool invoked your binary, deserializes the event into a normalized struct, and serializes your response back — in whatever format that tool expects.

Without polyhook — different format per tool 
// Claude Code
stdin:  "tool_name": "Bash",
        "tool_input": { "command": "..." }
stdout: "decision": "block"

// Cursor
stdin:  "type": "BeforeToolCall",
        "toolCall": { "name": "run_terminal_cmd" }
stdout: "action": "deny"

// Windsurf
stdin:  "event": "pre_tool",
        "tool": "run_command"
stdout: "allow": false

→ write a parser + serializer for each tool
With polyhook — one hook, every tool 
import { read, respond, block, approve }
  from "@polyhook/sdk";

// normalized HookEvent — any caller
const event = await read();

if (
  event.tool === "bash" &&
  /rm\s+-rf/.test(event.input?.command ?? "")
) {
  await respond(block("Blocked destructive command"));
} else {
  await respond(approve());
}

// runs unchanged on Claude Code, Cursor,
// Windsurf, Cline, Amp, Gemini CLI
Architecture

One WASM core. Every language.

Rust compiled to WebAssembly handles all detection, normalization, and serialization. Each language SDK is a thin shim — no logic reimplemented per language, no divergence possible.

AI tool ──stdin──▶ your binary │ polyhook.read() ← WASM: detect caller, normalize event │ your hook logic ← tool-agnostic, any language │ polyhook.respond() ← WASM: serialize to caller's format │ AI tool ◀─stdout── your binary

interface HookEvent { event: "tool:before" | "tool:after" | "session:start" | ... tool?: string // "bash", "write_file", "web_search", ... input?: Record<string, unknown> // tool:before only output?: Record<string, unknown> // tool:after only caller: "claude-code" | "cursor" | "windsurf" | "cline" | "amp" | ... sessionId: string }
Three response actions
approve()
Proceed with the operation unchanged.
block("reason")
Abort the operation. Your message is surfaced to the user.
modify({ command: "safe" })
Execute with a different input than the AI tool provided.

Design

Built on four constraints.

These aren't aspirational — they're enforced by the architecture.

⬡

One implementation

All detection and serialization lives in core. Language SDKs are thin wrappers over the same WASM binary — divergence is structurally impossible.

⚡

Runtime only

No code generation. polyhook runs inside your binary at hook invocation time. No build-step complexity, no generated files to maintain.

◎

Transparent fallback

Unknown caller → caller: "unknown", best-effort parse. Your hook keeps running — it does not crash on unrecognized tools.

⬜

Bring your own runtime

Each SDK ships polyhook.wasm and a thin host binding. Swap the WASM runtime if your platform requires it. See BINDINGS.md.
